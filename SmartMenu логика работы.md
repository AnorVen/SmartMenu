# Сервис 1.1 – AI-Orchestrator 1

1. Команда /generate  
   1. Запрашивает профиль (1.6) – бюджет, вес, транспорт, диету, аллергии, время готовки.  
   2. Запрашивает остатки (1.2) – что есть дома.  
   3. Запрашивает рецепты по остаткам (1.5) – список подходящих.  
   4. Запрашивает цены на недостающие ингредиенты (1.3).  
   5. Формирует промпт для LLM с жёсткими ограничениями (стоимость ≤ бюджет, вес ≤ max\_weight).  
   6. Получает JSON-меню, валидирует, при ошибке повторяет (макс. 2 раза).  
   7. Сохраняет меню в кэш (Redis) и возвращает боту.  
2. Пользователь отмечает пропуск (сегодня не будет ужина)  
   1. Загружает текущее меню.  
   2. Запрашивает актуальные остатки из 1.2 (учитывая уже купленное).  
   3. Определяет скоропортящиеся продукты из пропущенного блюда.  
   4. Формирует промпт для LLM: «Перераспредели эти ингредиенты на оставшиеся дни, минимизируй новые покупки».  
   5. Получает обновлённое меню (только изменённые дни).  
   6. Сохраняет изменения, возвращает предупреждения (если продукт рискует пропасть).  
3. Учёт размера семьи (MVP)  
   В профиле указано household\_size \= 3  
   1. При генерации меню умножает порции всех рецептов на household\_size. Бюджет и вес масштабируются: budget \= base\_budget \* household\_size \* 0.8 (экономия на масштабе).  
4. Отсутствие духовки (В профиле appliances без oven)  
   1. Добавляет в промпт: «Не используй рецепты, требующие духовки (запекание, выпечка)». При запросе рецептов из 1.5 передаёт фильтр exclude\_appliances: \["oven"\].  
5. Конкретный ингредиент / блюдо (Пользователь пишет: «хочу курицу в каждом обеде»)  
   1. Добавляет в промпт обязательное условие: «В каждый обед включи курицу». Если указано конкретное блюдо – сначала ищет его через 1.5, затем фиксирует в нужный день.  
6. Гости на неделе (Пользователь указал guests: {day: 3, count: 4})  
   1. При генерации меню увеличивает порции в день 3 на количество гостей. Предупреждает о превышении бюджета/веса, предлагает подтвердить.

Идеальный результат:

7. Асинхронное перепланирование по событиям  
   1. Подписывается на события InventoryChanged, MealSkipped, UserLocationChanged. При изменении остатков автоматически пересчитывает оставшиеся дни (без участия пользователя).  
8. Персональная модель питания  
   1. Использует fine-tuned LLM (на истории пользователя). Генерация меню занимает \<5 секунд. Учитывает предыдущие оценки блюд.  
9.  Zero-waste с гарантией  
   1. Символьный движок \+ LLM: алгоритм строит граф использования скоропортящихся продуктов, гарантируя, что ни один продукт не пропадёт (с учётом сроков годности).  
10. Интеграция с трекерами здоровья  
    1. Получает данные о физической активности (Garmin, Apple Health) и предлагает блюда для восполнения дефицита калорий или витаминов.

# Сервис 1.2 – Inventory Manager

1. Добавление продуктов после покупки  
   1. Принимает список продуктов (название, количество, цена).  
   2. Для каждого определяет категорию и скоропортящесть.  
   3. Устанавливает expiration\_date \= сегодня \+ дни\_годности (зелень – 2, мясо – 5, бакалея – 365).  
   4. Сохраняет запись в БД.  
2. Списание продуктов после готовки  
   1. Получает список ингредиентов блюда.  
   2. Ищет соответствующие продукты в инвентаре (по названию, ближайшие к истечению).  
   3. Помечает их как is\_consumed \= True.  
   4. Возвращает количество списанных позиций.  
3. Получение остатков для AI  
   1. Выбирает все активные продукты (is\_consumed=False).  
   2. Формирует текстовое описание: «У вас дома: молоко (200 мл, годно до 10.04), курица (300 г)…».  
   3. Возвращает JSON со списком и текстом.  
4. Отслеживание истекающих продуктов  
   1. Находит продукты, у которых expiration\_date \<= today \+ 3.  
   2. Для каждого пользователя формирует список.  
   3. Отправляет уведомление в бот (через API бота).  
5. Очистка просроченных  
   1. Находит продукты с expiration\_date \< today.  
   2. Устанавливает is\_consumed \= True.  
   3. Логирует количество утилизированного.

Идеальный результат

6. Частичное списание  
   1. Пользователь может указать, что использовано 150 г из 200 г курицы. Система создаёт новую запись с остатком 50 г и новым сроком годности (пересчёт).  
7. Интеграция с умным холодильником  
   1. Автоматическое получение данных о добавленных/изъятых продуктах через API холодильника (Samsung, LG).  
8. Распознавание по фото  
   1. Пользователь фотографирует холодильник – нейросеть определяет продукты, сервис автоматически добавляет их в инвентарь.  
9. Предсказание расхода  
   1. На основе истории потребления прогнозирует, когда закончится молоко, и добавляет в список покупок автоматически.

# Сервис 1.3 – Retail Data Aggregator

1. Получение цены продукта (Оркестратор запрашивает цену на "молоко" в "Магните")  
   1. Проверяет кэш (Redis).  
   2. Если нет – ищет в БД пользовательских цен (агрегированных).  
   3. Если нет – вызывает парсер Магнита (симулированный или реальный).  
   4. Возвращает цену (или None, если не найдена).  
2. Ручной ввод цены пользователем   
   Пользователь пишет «цена молоко 70 руб, Магнит»  
   1. Сохраняет запись в таблицу price\_records с source="user", user\_id.  
   2. Обновляет агрегированную цену (медиану по всем пользователям).  
   3. Инвалидирует кэш.  
3. Сравнение цен для списка покупок  
   1. Для каждого продукта определяет лучшую цену по разным сетям.  
   2. Возвращает словарь: продукт → магазин → цена.

Идеальный результат

4. Прямые API ритейлеров  
   1. Интеграция с партнёрскими API (Магнит, Пятёрочка) – получение актуальных цен, остатков на полке, персональных скидок по карте лояльности.  
5. OCR чеков (краудсорсинг)  
   1. Пользователь загружает фото чека → сервис распознаёт позиции и цены → пополняет общую базу (анонимно).  
6. Предсказание будущих скидок  
   1. На основе исторических данных (сезонность, дни недели) система сообщает: «Обычно в пятницу скидка на курицу 20%, подожди до завтра».  
7.  Сравнение с доставкой  
   1. Показывает, где дешевле: купить в магазине (с учётом времени и транспорта) или заказать с доставкой (Яндекс Лавка, СберМаркет).

# Сервис 1.4 – Logistics & Route Planner

1.  Построение маршрута по магазинам (Пользователь запрашивает маршрут для списка покупок)  
   1. Получает список товаров (с весом и конкретным магазином).  
   2.   
2. Проверка веса (Пользователь вводит список покупок)  
   1. Суммирует вес всех товаров.  
   2. Если превышает max\_weight\_kg, предлагает разбить на два похода (или выводит предупреждение).

Идеальный результат

3. Построение маршрута  
   Вход:  
- Список магазинов с товарами (из Retail Aggregator)  
- Текущее местоположение пользователя (lat, lon)  
- Транспорт (пешком / автобус / авто)  
- Максимальный вес за поход  
  1. Группируем товары по магазинам (один магазин может быть посещён несколько раз, если вес превышает лимит).  
  2. Для каждого магазина определяем координаты (из локальной базы или через гео-сервис).  
  3. Решаем задачу коммивояжёра (TSP) с жадной эвристикой:

     \- Старт из текущей точки пользователя.

     \- На каждом шаге выбираем ближайший непосещённый магазин.

  4. Для каждого перехода вычисляем расстояние и время через Maps API.  
  5. Если общий вес всех товаров \> max\_weight:

     \- Разбиваем на несколько походов (алгоритм рюкзака).

     \- Каждый поход — отдельный маршрут в разные дни.

  6. Сохраняем маршрут в БД с привязкой к дням.  
4. Гео-уведомления

   Триггер: Пользователь движется по маршруту (фоновая геолокация в боте).

   1. Каждые N секунд (например, 30\) бот получает текущие координаты.  
   2. Вычисляется расстояние до каждого запланированного магазина в маршруте на сегодня.  
   3. Если расстояние \< 50 м и магазин ещё не отмечен как «посещённый сегодня»:

      \- Бот отправляет push-уведомление: «Вы рядом с \[магазин\]. Ваш список: \[товары\]».

      \- Отмечаем магазин как «уведомление отправлено» на сегодня (чтобы не спамить).

   4. Если пользователь заходит в магазин, не запланированный на сегодня:

      \- Система проверяет, есть ли в этом магазине товары из будущих дней по более низкой цене.

      \- Если да — предлагает купить сейчас.

   5. После выхода из магазина (гео-зона \> 100 м) система помечает, что магазин посещён.

# Сервис 1.5 – Recipe & Content Service (Рецепты)

1. Поиск рецептов по остаткам  
   1. Ищет в БД рецепты, в которых ≥50% ингредиентов совпадают с остатками.  
   2. Сортирует по проценту совпадения.  
   3. Возвращает до 10 рецептов.  
2. Адаптация рецепта под доступные ингредиенты  
   1. Анализирует недостающие ингредиенты.  
   2. Ищет замены (через LLM или таблицу простых замен).  
   3. Возвращает адаптированные инструкции, список замен и список всё ещё недостающих ингредиентов.  
3.  Получение рецепта по ID  
   1. Возвращает полную информацию: название, ингредиенты, пошаговые инструкции, время готовки, сложность.  
4. Фильтрация по доступной технике  
   1. Оркестратор передаёт exclude\_appliances: \["oven"\]  
   2. Исключает рецепты, у которых в поле required\_appliances есть oven.

Идеальный результат

5. Векторный поиск (семантический)  
   1. Пользователь пишет «что-то острое, быстрое, с курицей» – сервис находит рецепты по смыслу (нейросетевая эмбеддинги).  
6. Генерация новых рецептов через LLM  
   1. Если ни один рецепт не подходит, LLM генерирует новый на основе остатков и предпочтений.  
7. Загрузка пользовательских рецептов  
   1. Пользователь может добавить свой рецепт, указав ингредиенты, инструкции, фото. Сервис сохраняет и индексирует.  
8. Автоматическая адаптация под диету  
   1. Рецепт с глютеном автоматически конвертируется в безглютеновый (замена муки, соевого соуса и т.д.).

# Сервис 1.6 – User Profile & Constraints (Профиль)

1. Создание профиля  
   1. Проверяет, существует ли профиль с таким user\_id.  
   2. Если нет – создаёт запись с дефолтными значениями (бюджет 2000, вес 3 кг, пешком, всёядный, household\_size=1, appliances=\["stove","oven","microwave"\]).  
2. Получение ограничений для оркестратора (Оркестратор запрашивает GET /constraints)  
   1. Возвращает только поля, необходимые для генерации меню: бюджет, вес, транспорт, диету, аллергии, нелюбимые продукты, время готовки, household\_size, appliances.  
3.  Обновление профиля (размер семьи)  
   1. Пользователь вводит семья 3;  
   2. Обновляет поле household\_size.  
4. Обновление профиля (отсутствие духовки)  
   1. Обновляет поле appliances \= \["stove","microwave"\].  
5. Добавление гостей на неделю (Пользователь вводит гости среда 4\)  
   1. Сохраняет в поле guests\_schedule запись {day: 3, count: 4}. При генерации следующего меню оркестратор учтёт.  
6. Учёт конкретного пожелания (Пользователь пишет «хочу курицу»)  
   1. Это НЕ хранится в профиле, а передаётся разово в запросе на генерацию. Сервис 1.6 не участвует.

Идеальный результат

7.  Мультипрофиль для семьи  
   1. Каждый член семьи может иметь свои аллергии, диету, предпочтения. Общий бюджет и инвентарь – семейные.  
8. История изменений  
   1. Аудит всех изменений профиля (кто, когда, что изменил).  
9. Интеграция с фитнес-трекерами  
   1. Автоматический расчёт рекомендуемой калорийности на основе активности (шаги, тренировки).  
10.  Анализ поведения и рекомендации  
    1. Система предлагает увеличить лимит веса, если пользователь часто превышает, или уменьшить бюджет, если экономит.

# Взаимодействие сервисов (логика запросов)

Алёна – студентка, живёт одна, бюджет 2000 руб. на неделю, максимальный вес покупок 3 кг, передвигается пешком, у неё нет духовки. Она хочет, чтобы в каждом обеде была курица, а на среду у неё запланированы гости (4 человека). Алёна пишет боту в Telegram.

1. Настройка профиля  
   Бот обращается к сервису 1.6 (User Profile). Тот проверяет, существует ли профиль Алёны. При первом обращении создаётся запись с дефолтными значениями. Затем бот передаёт уточнения: бюджет 2000 руб., вес 3 кг, транспорт «пешком», диета «всеядная», отсутствие духовки (поле appliances без oven), размер семьи 1\. Сервис сохраняет все ограничения. Алёна также указывает, что на среду (день 3\) придут 4 гостя – сервис сохраняет это в поле guests\_schedule  
2. Генерация меню на неделю  
   Алёна отправляет команду /generate с пожеланием «хочу курицу в каждом обеде».  
   1. Оркестратор получает задачу  
      Бот передаёт запрос в сервис 1.1 (AI-Orchestrator). Оркестратор начинает собирать данные:  
      1. Запрос к сервису 1.6 – получает все ограничения: бюджет, вес, транспорт, диету, аллергии (если есть), отсутствие духовки, размер семьи, расписание гостей.  
      2. Запрос к сервису 1.2 (Inventory) – узнаёт, какие продукты уже есть дома. В начале недели инвентарь пуст.  
      3. Запрос к сервису 1.5 (Recipe Service) – передаёт список остатков (пусто) и фильтр «исключить рецепты, требующие духовки». Сервис возвращает подборку рецептов, подходящих для студента (быстрые, простые).  
      4. Запрос к сервису 1.3 (Retail Aggregator) – для каждого недостающего ингредиента из найденных рецептов запрашивает цену в ближайших магазинах (Магнит, Пятёрочка). Сервис возвращает цены (сначала из кэша, затем из парсинга или базовых значений).  
   2. Оркестратор формирует промпт для LLM  
      Он объединяет все данные: бюджет 2000 руб., максимальный вес 3 кг, отсутствие духовки, требование «курица в каждом обеде», гости в среду (увеличить порции на 4 персоны). Промпт требует 3\. строгого соблюдения бюджета и веса, а также JSON-формата с меню на 7 дней (21 приём пищи).  
   3. Оркестратор вызывает LLM  
      LLM генерирует меню. Оркестратор валидирует результат: общая стоимость не должна превышать 2000 руб., общий вес – 3 кг. Если превышение – повторяет запрос с требованием уменьшить. После успешной валидации меню сохраняется в кэше (Redis) и возвращается боту.  
   4. Бот показывает меню  
      Telegram-бот форматирует меню и отправляет Алёне. В меню указано, что в среду порции увеличены для гостей, а все обеды содержат курицу.  
3. Построение маршрута и списка покупок  
   Алёна нажимает кнопку «Список покупок». Бот запрашивает текущее меню у сервиса 1.1. Оркестратор возвращает агрегированный список продуктов с указанием количества и магазинов, где их выгоднее купить (данные от сервиса 1.3).  
   Алёна нажимает «Построить маршрут» и отправляет свою геолокацию. Бот передаёт список покупок, геопозицию и ограничение по весу (3 кг) в сервис 1.4 (Logistics & Route Planner).  
   1. Сервис 1.4:  
      Запрашивает у сервиса 1.3 координаты магазинов, фигурирующих в списке.  
        
      Распределяет товары по магазинам, проверяя, чтобы вес покупок в каждом магазине не превышал 3 кг (если превышает – разбивает на два похода или выдаёт предупреждение).  
        
      Строит оптимальный порядок посещения магазинов (жадный алгоритм, расстояние по прямой или через простой вызов картографического API).  
        
      Возвращает боту маршрут: последовательность магазинов с адресами, список того, что купить в каждом, общее расстояние и предполагаемое время.  
      Бот отображает маршрут Алёне.  
4. Поход в магазин и обновление инвентаря  
   Алёна идёт по маршруту, покупает всё. После возвращения она нажимает «Купил всё». Бот отправляет список купленных продуктов в сервис 1.2 (Inventory). Сервис добавляет каждый продукт в базу, автоматически проставляя срок годности в зависимости от категории (например, зелень – 2 дня, курица – 5 дней). Теперь эти продукты числятся как «дома».  
5. Приготовление блюд и списание продуктов  
   В течение недели Алёна готовит по меню. После каждого приготовленного блюда она отмечает его в боте. Бот вызывает сервис 1.2 с запросом списать ингредиенты этого блюда. Сервис находит соответствующие продукты в инвентаре (по названию, с приоритетом на те, у которых срок истекает раньше) и помечает их как is\_consumed. Если продукт использован не полностью (в MVP – только целиком), он просто исчезает из активного инвентаря.  
6. Обработка исключения – пропуск ужина в четверг  
   В четверг вечером Алёна уходит в гости и не может приготовить запланированный ужин. Она открывает бот, выбирает приём пищи «ужин четверг» и нажимает «Пропустить».

Бот отправляет событие в сервис 1.1. Оркестратор:

* Загружает текущее меню.  
* Запрашивает актуальные остатки из сервиса 1.2 (теперь там есть курица, зелень и другие продукты).  
* Определяет, какие ингредиенты из пропущенного ужина уже куплены и могут пропасть.  
* Формирует новый промпт для LLM с просьбой перераспределить эти продукты на оставшиеся дни (пятница, суббота, воскресенье) без дополнительных покупок, где возможно.  
* Получает обновлённое меню (только изменённые дни), сохраняет его.  
* Если какой-то продукт (например, свежая зелень) рискует пропасть, оркестратор добавляет предупреждение.

Бот отправляет Алёне обновлённое меню и предупреждение: «Зелень нужно использовать до завтра, я добавил её в завтрашний сэндвич».

7. Уведомление об истекающих продуктах  
   На следующий день утром сервис 1.2 по расписанию (cron) проверяет, какие продукты истекают в ближайшие 3 дня. Он находит, что молоко истекает через 1 день. Сервис отправляет уведомление в Telegram-бота (через прямой вызов API или через общую базу). Бот присылает Алёне сообщение: «У вас заканчивается срок годности молока (остался 1 день). Используйте его в завтрашней овсянке или добавьте в кофе». Алёна может нажать кнопку «Предложить рецепт» – тогда бот запросит сервис 1.5 и вернёт рецепт с использованием молока.  
8. Завершение цикла  
   В конце недели Алёна получила сводку от сервиса 1.1 (через бота): сколько приемов пищи приготовлено, сколько денег сэкономлено, сколько продуктов не выброшено. Сервис 1.6 обновил статистику пользователя (стрейк, количество приготовленных блюд). Все шесть сервисов работали синхронно, обмениваясь REST-запросами, без асинхронных брокеров, но с простым кэшированием (Redis). Каждый сервис выполнял свою узкую функцию, а оркестратор координировал их для достижения цели – персонализированного, безотходного и логистически удобного плана питания.

# ❌ старыйСервис 1.1 – AI-Orchestrator

Алгоритм ядра

1. Zero-waste: использование остатков  
   **вход:** список остатков (продукты+количество+срок годности)  
   **выход:** список блюд, которые максимально используют эти остатки  
   1. Сгруппировать остатки по категориям:  
      Скоропортящиеся (зелень, молоко, мясо)  
      Овощи/фрукты (приоритет 2\)  
      Бакалея (приоритет 3\)  
   2. Для каждого скоропортящегося продукта:  
      Определить срок до порчи  
      Запланировать его использование в D блюдах  
   3. При генерации меню добавить в промпт жесткое требование: ты обязан использовать указанные продукты до указанных дат  
   4. После генерации проверить, что все скоропортящиеся продукты действительно использованы. Если нет — регенерировать с усиленным требованием.  
      **Пример:**  
      Если в остатках есть укроп (пучок, портится через 2 дня):  
      Он должен появиться минимум в 2 разных блюдах (например, в салате и супе)  
      Блюда должны быть запланированы на сегодня и завтра  
2. Семейный профиль (несколько человек)  
   1. Структура  
      Семейный аккаунт (group\_id)  
      ├── Общий бюджет (3000 руб/неделя)  
      ├── Общий инвентарь (общие продукты)  
      ├── Список участников:  
      │   ├── Алёна (роль: администратор)  
      │   │   ├── предпочтения: всеядный, время готовки 30 мин  
      │   │   └── аллергии: нет  
      │   └── Парень (роль: участник)  
      │       ├── предпочтения: всеядный  
      │       └── аллергии: орехи  
      └── Общие ограничения:  
          ├── макс. вес покупок: 5 кг (совместный поход)  
          ├── транспорт: пешком  
          └── доступная техника: без духовки  
   2. Логика генерации меню для семьи  
      1. При генерации меню AI-Orchestrator учитывает:  
         \- Аллергии всех членов семьи (орехи исключены)  
         \- Диетические предпочтения (если разные — ищем компромисс: блюда, которые подходят всем)  
         \- Время готовки (среднее или максимальное по желанию)  
      2. Порции рассчитываются на количество человек (2).  
      3. Если один из членов семьи пропускает приём пищи (в гостях):  
         \- Система уменьшает порцию на 1, оставшиеся ингредиенты перераспределяет.  
      4. Статистика ведётся общая (сэкономлено на семью).  
   3. Разделение ролей  
      1. Администратор (Алёна) может менять настройки, генерировать меню, подтверждать покупки.  
      2. Участник (парень) может отмечать, что он купил, приготовил или пропустил приём пищи, но не менять бюджет.

3. Перепланирование после пропуска приема пищи  
   **Вход:** ID пользователя, день и тип пропущенного приёма пищи  
   **Выход**: обновленное меню и список покупок

   1. Получить текущее меню из БД.  
   2. Найти пропущенный прием пищи, извлечь список ингредиентов.  
   3. Разделить ингредиенты на:  
      \- Уже купленные (есть в Inventory Manager)  
      \- Не купленные (ещё в списке покупок)  
   4. Для некупленных ингредиентов:  
      \- Удалить их из списка покупок (если они не используются в других блюдах)  
   5. Для купленных скоропортящихся ингредиентов:  
      \- Определить срок годности каждого  
      \- Если срок годности истекает в ближайшие 2 дня:  
              \* Найти рецепты из Recipe Service, которые используют эти ингредиенты  
              \* Выбрать рецепт с минимальным временем готовки и подходящий по диете  
              \* Предложить заменить одно из будущих блюд (на сегодня или завтра)  
   6. Если подходящей замены нет:  
      \- Отправить предупреждение: "Ингредиенты X, Y могут испортиться. Используйте их в другом блюде вручную"  
   7. Обновить меню и список покупок.  
   8. Отправить пользователю подтверждение с изменениями.  
        
4. Ограничения по устройствам (отсутствие духовки)  
   1. Хранение в профиле  
      Добавляется новое поле: available\_appliances: list\[string\]  
      Возможные значения: stove, microwave, oven, airfryer, slow\_cooker, no\_oven.  
   2. Логика фильтрации рецептов  
      При поиске рецептов (Recipe Service) и генерации меню:  
      1. Из базы рецептов выбираются только те, чьи требования к оборудованию ⊆ доступной технике.  
      2. Если у пользователя нет духовки:  
         \- Исключаются рецепты, требующие запекания (если нет альтернативного способа).  
         \- LLM может адаптировать рецепт: «запечь в духовке» → «обжарить на сковороде под крышкой».  
      3. При адаптации рецепта (через LLM) система явно указывает замену способа приготовления.

	

# ❌ Обработки ошибок

| Компонент | Ошибка | Действие | Fallback |
| :---- | :---- | :---- | :---- |
| AI-Orchestrator | LLM API не отвечает 30 сек | Повтор 1 раз | Вернуть шаблонное меню (заранее заготовленное) |
| AI-Orchestrator | LLM вернул невалидный JSON | Повтор с temperature=0.3 | Вернуть fallback меню |
| AI-Orchestrator | Бюджет превышен после генерации | Добавить в промпт "Уменьши стоимость\!" и регенерировать (макс 2 раза) | Урезать список покупок, удалив самый дорогой не essential товар |
| Logistics | Yandex Maps API ошибка (429, 500\) | 2 повтора с экспоненциальной задержкой | Рассчитать расстояние по прямой (геодезическое) |
| Logistics | Нет информации о магазине | Использовать "ближайший магазин неизвестной сети" | По умолчанию: расстояние 1 км, вес не учитывать |
| Retail Aggregator | Парсинг сайта не удался | Логировать ошибку | Вернуть последнюю цену из кэша или цену от пользователя |
| Inventory | Пользователь списывает продукт, которого нет в инвентаре | Игнорировать, логировать предупреждение | Ничего не делать |
| Telegram Bot | Пользователь отправил некорректную команду | Ответить с подсказкой | Показать клавиатуру с доступными действиями |
