# Архитектура SmartMenu

## Цель продукта

SmartMenu помогает пользователю составить здоровое меню на неделю, сократить пищевые отходы, уложиться в бюджет и получить удобный маршрут закупок с учетом веса покупок, транспорта, магазинов и личных ограничений.

## Принцип декомпозиции

Проект разделен по доменным границам. Каждый сервис владеет своей областью данных и может развиваться независимо. Для MVP сервисы общаются синхронно по REST. Redis используется для кэша меню и цен, Postgres — как основное хранилище.

## Сервисы

### `ai-orchestrator`

Главный координатор сценариев:

- команда `/generate`;
- сбор профиля из `user-profile-service`;
- получение остатков из `inventory-manager`;
- подбор рецептов через `recipe-content-service`;
- запрос цен через `retail-data-aggregator`;
- генерация и валидация меню;
- перепланирование после пропуска приема пищи;
- подготовка списка покупок.

База: `smartmenu_ai_orchestrator`.

### `inventory-manager`

Отвечает за продукты пользователя:

- добавление продуктов после покупки;
- расчет срока годности;
- списание ингредиентов после готовки;
- выдача активных остатков для AI;
- поиск истекающих продуктов;
- уведомления через `telegram-bot-gateway`.

База: `smartmenu_inventory`.

### `retail-data-aggregator`

Единая точка работы с ценами:

- получение цены продукта;
- ручной ввод цены пользователем;
- агрегация пользовательских цен;
- сравнение цен по магазинам;
- кэширование цен в Redis.

База: `smartmenu_retail`.

### `logistics-route-planner`

Планирует закупки:

- проверяет вес покупок;
- разбивает покупки на несколько походов;
- строит маршрут по магазинам;
- учитывает транспорт, обычные маршруты пользователя и магазины по пути;
- в будущем может работать с геозонами и push-уведомлениями.

База: `smartmenu_logistics`.

### `recipe-content-service`

Владеет рецептами и контентом:

- поиск рецептов по остаткам;
- фильтрация по технике, диете и аллергиям;
- подбор замен ингредиентов;
- хранение пользовательских рецептов;
- в будущем — семантический поиск через эмбеддинги.

База: `smartmenu_recipe`.

### `user-profile-service`

Хранит пользовательские ограничения:

- бюджет;
- максимальный вес покупок;
- транспорт;
- household/family profile;
- аллергии и нежелательные продукты;
- доступная техника;
- расписание гостей;
- статистика пользователя.

База: `smartmenu_user_profile`.

### `telegram-bot-gateway`

Внешний пользовательский шлюз:

- принимает Telegram webhook;
- переводит команды пользователя в вызовы внутренних сервисов;
- отправляет уведомления;
- не владеет доменной логикой меню, цен, рецептов или инвентаря.

На старте сервис можно оставить stateless. Если понадобится история диалогов, лучше добавить отдельную базу `smartmenu_bot_gateway`.

#### Webhook-и и события бота

В MVP `telegram-bot-gateway` принимает один внешний webhook от Telegram:

- `POST /telegram/webhook` — основной вход для Telegram updates.

Внутренние сервисы могут отправлять уведомления пользователю через:

- `POST /notifications` — отправка текста пользователю по `chatId`.

Типы Telegram updates, на которые реагирует бот:

- `message.text` с командой `/start` — старт onboarding и сбор профиля.
- `message.text` с командой `/profile` — повторное заполнение профиля.
- `message.text` с командой `/generate` — запрос меню на неделю.
- `message.text` с командой `/route` — запрос геолокации для маршрута.
- `message.text` с командой `/inventory` — показ текущих остатков.
- `message.text` с командой `/summary` — недельная сводка по приготовленным блюдам, экономии и спасенным продуктам.
- `message.text` с командой `/help` — список доступных команд.
- `message.text`, начинающийся с `купила` или `купил` — добавление покупки в `inventory-manager`.
- `message.text`, начинающийся с `цена` — ручной ввод цены в `retail-data-aggregator`.
- `message.text` с фразами `пропускаю` или `пропустить` — событие пропуска приема пищи для `ai-orchestrator`.
- `message.text` с фразами `гости` или `придут` — планирование гостей через `user-profile-service` и пересчет меню через `ai-orchestrator`.
- `message.text`, начинающийся с `хочу` — пожелание конкретного блюда или ингредиента для `ai-orchestrator`.
- `message.text`, начинающийся с `замени` или `заменить` — замена ингредиента через `recipe-content-service` и пересчет меню.
- `message.text` с фразой `приготов` — списание ингредиентов через `inventory-manager`.
- `message.text` с фразой `предложи рецепт` — подбор рецепта из остатков через `recipe-content-service`.
- `message.text` с фразами `сводка` или `итоги недели` — недельная сводка.
- `message.text` с фразами `я в магазине`, `зашла в магазин`, `зашел в магазин` — проверка незапланированного магазина через `logistics-route-planner`.
- `message.photo` — прием фото чека как источник цен для будущего OCR.
- `message.location` — построение маршрута через `logistics-route-planner`.
- `callback_query.data = generate_menu` — генерация меню по кнопке.
- `callback_query.data = build_route` — запрос маршрута по кнопке.
- `callback_query.data = show_inventory` — показ инвентаря по кнопке.
- `callback_query.data = restart_profile` — перезапуск анкеты.
- `callback_query.data = skip_meal:<meal>` — перепланирование после пропуска приема пищи.
- `callback_query.data = mark_cooked` — отметка блюда приготовленным.
- `callback_query.data = suggest_recipe:expiring` — рецепт из истекающих продуктов.
- `callback_query.data = weekly_summary` — недельная сводка.

Данные, которые бот собирает при первом запуске:

- недельный бюджет;
- максимальный вес покупки;
- количество человек;
- транспорт;
- комфортное время готовки;
- удобные слоты готовки;
- доступная техника;
- диета;
- аллергии и нежелательные продукты;
- удобные районы или маршруты покупок;
- количество приемов пищи в день.
- особенности семьи или участников.

После завершения анкеты бот отправляет профиль в `user-profile-service`, а затем предлагает сгенерировать меню или построить маршрут.

## Поток MVP

1. Пользователь пишет `/start` в Telegram.
2. `telegram-bot-gateway` создает или обновляет профиль через `user-profile-service`.
3. Пользователь просит меню на неделю.
4. `telegram-bot-gateway` вызывает `ai-orchestrator`.
5. `ai-orchestrator` собирает ограничения, остатки, рецепты и цены.
6. `ai-orchestrator` вызывает LLM, валидирует бюджет, вес, аллергию и доступную технику.
7. Пользователь получает меню и список покупок.
8. Для маршрута бот вызывает `logistics-route-planner`.
9. После покупки бот отправляет продукты в `inventory-manager`.
10. После готовки бот списывает ингредиенты через `inventory-manager`.
11. При пропуске приема пищи `ai-orchestrator` перепланирует оставшиеся дни.

## Инфраструктура

- `docker-compose.yml` поднимает все сервисы, Postgres и Redis.
- Postgres запускается с init-скриптом `infrastructure/postgres/init/01-create-databases.sql`.
- Каждый сервис имеет отдельный Dockerfile и образ `smartmenu/<service>:local`.
- В MVP используется один Postgres-контейнер с несколькими базами. Это проще для локальной разработки, но сохраняет логическую изоляцию доменов.

## Почему один Postgres-контейнер

Для локальной разработки отдельный контейнер Postgres на каждый сервис усложнит запуск, порты и потребление ресурсов. Текущий вариант создает отдельные базы внутри одного Postgres. При переходе к production каждую базу можно вынести в отдельный managed Postgres или отдельный кластер без изменения границ сервисов.

## Prisma и миграции

Для сервисов с собственной Postgres-базой выбран Prisma:

- `ai-orchestrator`;
- `inventory-manager`;
- `retail-data-aggregator`;
- `logistics-route-planner`;
- `recipe-content-service`;
- `user-profile-service`.

В каждом из этих сервисов есть:

- `prisma/schema.prisma` — доменная схема базы;
- `npm run db:generate` — генерация Prisma Client;
- `npm run db:migrate` — создание и применение dev-миграции;
- `npm run db:deploy` — применение миграций в production-like окружении;
- `npm run db:push` — быстрая синхронизация схемы для локальных прототипов.

`telegram-bot-gateway` пока не использует Prisma, потому что он stateless и не владеет своей базой.

## Что добавить следующим этапом

- Контракты API через OpenAPI.
- Валидацию входящих данных через Zod.
- Общий correlation id для трассировки запросов.
- Очередь событий для идеального сценария: `InventoryChanged`, `MealSkipped`, `UserLocationChanged`.
- Отдельный worker для cron-задач Inventory.
- Интеграции с LLM, картами и ритейлерами через adapter layer.
