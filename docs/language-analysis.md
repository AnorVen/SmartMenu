# Анализ языков для сервисов

Пользовательское требование: проекты пишем на Node.js. Поэтому каркас подготовлен на Node.js + TypeScript. Ниже — инженерный анализ, какой язык был бы оптимальным для каждого сервиса, если выбирать без этого ограничения.

## Итоговая рекомендация для MVP

Для MVP лучше оставить все сервисы на Node.js/TypeScript:

- одна команда и один стек;
- быстрее разработка Telegram-бота, REST API и интеграций;
- проще Docker-сборка;
- строгая типизация TypeScript снижает риск ошибок;
- большинство сервисов I/O-bound, а не CPU-bound.

## По сервисам

| Сервис | Текущий стек | Идеальный стек при свободном выборе | Почему |
|---|---|---|---|
| `telegram-bot-gateway` | Node.js/TypeScript | Node.js/TypeScript | Telegram SDK, webhook/API, много I/O, быстрый цикл разработки. |
| `ai-orchestrator` | Node.js/TypeScript | Node.js/TypeScript или Python | Node.js удобен для координации REST-вызовов. Python сильнее, если внутри появится тяжелая ML-логика, но LLM-вызовы как внешний API хорошо ложатся на Node.js. |
| `inventory-manager` | Node.js/TypeScript | Node.js/TypeScript или Go | CRUD, транзакции, частичное списание, фоновые проверки сроков годности и уведомления. Node.js достаточно для MVP. Go может быть лучше на этапе роста: компактный сервис, низкое потребление памяти, строгая типизация и хорошая конкурентность для фоновых задач. |
| `retail-data-aggregator` | Node.js/TypeScript | Python или Node.js/TypeScript | Для парсинга, OCR и data pipelines Python богаче библиотеками. Для API-агрегатора и кэша Node.js подходит отлично. |
| `logistics-route-planner` | Node.js/TypeScript | Go, Rust или Python | Если маршрутизация станет алгоритмически тяжелой, Go/Rust дадут производительность. Если опираться на внешние карты, Node.js достаточно. |

| `user-profile-service` | Node.js/TypeScript | Node.js/TypeScript или Go | Профильный CRUD-сервис с понятной схемой: бюджет, семья, аллергии, техника, гости и ограничения. Node.js полностью подходит для MVP. Go может быть хорошим production-вариантом для небольшого надежного сервиса с Postgres. |

## Детальные выводы

### Node.js/TypeScript

Лучший выбор для текущего MVP. Почти все сервисы ждут сеть, БД, Redis, LLM, карты или Telegram. Для такой нагрузки Node.js эффективен и прост. TypeScript нужен обязательно, потому что продукт быстро обрастет контрактами: меню, ингредиенты, ограничения, гости, семья, аллергии, техника.

### Python

Лучше всего подходит для будущих частей:

- OCR чеков;
- распознавание продуктов по фото;
- семантический поиск рецептов;
- обучение персональных моделей;
- прогноз скидок;
- сложный zero-waste optimizer с ML/OR-инструментами.

Рекомендация: не переводить весь backend на Python, а позже добавить отдельные ML/data-сервисы.

### Go

Может быть полезен для `inventory-manager`, `user-profile-service` и `logistics-route-planner`. Для первых двух Go дает простой production-деплой, низкое потребление памяти и надежную работу с Postgres/фоновой обработкой. Для `logistics-route-planner` Go особенно полезен, если внутри появятся собственные алгоритмы маршрутизации, геоиндексы и высокая нагрузка. Для MVP переводить эти сервисы на Go преждевременно: единый Node.js/TypeScript-стек быстрее.

### Rust

Имеет смысл только для критичных вычислительных модулей: оптимизация маршрутов, zero-waste solver, тяжелая обработка графов. Для раннего продукта стоимость разработки выше пользы.

### Kotlin/Java

Хороший выбор для больших enterprise-систем с сильными транзакционными правилами и уже существующей JVM-инфраструктурой. Для SmartMenu Kotlin/Java сейчас избыточны: если понадобится второй backend-язык для небольших надежных сервисов, практичнее смотреть в сторону Go.

## Финальная позиция

Сейчас все сервисы правильно делать на Node.js/TypeScript. Исключения стоит добавлять позже точечно:

- Python service для ML/OCR/embeddings.
- Go service для `inventory-manager`, `user-profile-service` или тяжелой маршрутизации, если появится реальная production-нагрузка.
- Rust service для критичных вычислительных модулей вроде zero-waste solver или тяжелой обработки графов.
- Все внешние API и пользовательские сценарии оставить в Node.js.
